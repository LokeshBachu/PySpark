Chapter 1 · Introduction

What is PySpark?

1.1 The Big Picture

Imagine you have 500 GB of user clickstream data but your laptop only has 16 GB of RAM. PySpark solves exactly this problem.

PySpark is the Python API for Apache Spark — a distributed computing engine that divides work across multiple machines in a cluster, each handling a portion of the data simultaneously.

Analogy Sorting 1 million books alone is slow. Hire 100 people, give each 10,000 books, sort in parallel, merge results = fast and scalable. That is distributed computing.

1.2 Pandas vs PySpark

FeaturePandasPySpark
Data SizeUp to ~10 GB (single machine)Petabytes (cluster)
ProcessingEager (runs immediately)Lazy (builds plan first)
ExecutionSingle machineDistributed
Use CaseData analysis, small ETLBig data ETL, ML at scale
Learning CurveEasierRequires distributed concepts
Speed — small dataFaster (no overhead)Slower (cluster overhead)
Speed — big dataCrashes / very slowScales linearly

1.3 Lazy Evaluation — The Key Concept

PySpark does not execute code line-by-line. It builds a DAG (Directed Acyclic Graph) and executes only when an action is called.

TypeBehaviourExamples
TransformationsLazy — extend the plan, do NOT executefilter() select() groupBy() join() withColumn()
ActionsEager — execute the full plan and return resultsshow() count() collect() write()
pythonlazy_eval.py
from pyspark.sql import SparkSession
from pyspark.sql.functions import col

spark = SparkSession.builder.appName("Demo").getOrCreate()

data = [("Alice",30,"HR"),("Bob",22,"IT"),("Charlie",28,"IT"),
        ("David",35,"Finance"),("Eve",24,"HR"),("Frank",40,"Finance")]
df = spark.createDataFrame(data, ["name","age","department"])

# Transformations — nothing executes yet
filtered = df.filter(col("age") > 25)
grouped  = filtered.groupBy("department").count()

# Action — Spark executes the full plan NOW
grouped.show()
Output
+----------+-----+ |department|count| +----------+-----+ | Finance| 2| | IT| 1| | HR| 1| +----------+-----+
Catalyst Optimizer Lazy evaluation lets Spark's Catalyst Optimizer reorder operations, push filters down, and skip unneeded columns — before any computation starts.

The Genesis of Spark

The Big Data Challenge

Increased consumer traffic, a variety of new forms of data, and greater computational demands required more storage and better performance. Traditional data storage methods — including Relational Database Management Systems (RDBMSs) and imperative programming techniques — were unable to handle enormous amounts of data and their processing.

The core problems that needed to be solved were:

  • Data collection and ingestion — ingesting large volumes from diverse sources
  • Data storage and management — storing petabytes reliably
  • Data processing and transformation — running complex computations at scale
  • Data access and retrieval — querying and serving results quickly

Google's White Papers

Google was the first to overcome these problems and published a series of influential white papers that laid the theoretical foundation for modern big-data systems:

YearPaperSolves
2003Google File System (GFS)Distributed, fault-tolerant storage
2004MapReduceParallel data processing at scale
2006BigtableStructured storage over distributed file systems

These white papers were highly appreciated by the open-source community and served as the inspiration for the design and development of a comparable open-source implementation — Hadoop.

Hadoop — The Open-Source Response

Hadoop is an open-source software framework for storing and processing large amounts of data in a distributed computing environment. Key characteristics:

  • Based on the MapReduce programming model, allowing parallel processing of large datasets
  • Framework written in Java
  • Designed to start small and expand as you grow — from a few nodes to thousands
  • Allows storage capacity of hundreds to thousands of computers, used as a unified storage system (HDFS)
  • Fault-tolerant by design — data is replicated across nodes

Hive — SQL on Hadoop

Many solutions were built on top of Hadoop. The most widely adopted was Apache Hive, which offered SQL-like capabilities on Hadoop data:

  • Create Databases, Tables, and Views
  • Run SQL Queries over HDFS data

Together, Hadoop + Hive became very popular. But significant problems remained:

ProblemDetail
PerformanceHive SQL queries were much slower than RDBMS SQL — every job wrote to disk
Ease of developmentWriting MapReduce programs was difficult and verbose
Language supportMapReduce was originally Java-only
Storage costHDFS on-premise storage was expensive compared to cloud storage
Resource managementOnly YARN container support — no Mesos, Docker, or Kubernetes

Hadoop left a lot of scope for improvement — and as a result, Apache Spark came into existence.

Hadoop vs Spark — Head to Head

AspectHadoop MapReduceApache Spark
Storage modelDisk after every stageIn-memory (RAM)
SpeedBaselineUp to 100× faster (iterative)
APIJava only (originally)Python, Scala, Java, R, SQL
StreamingBatch onlyBatch + real-time streaming
ML supportMahout (limited)MLlib built-in
Graph processingExternal toolsGraphX built-in

Spark Timeline

  • 2009 — Created at UC Berkeley AMPLab
  • 2010 — Open-sourced under BSD
  • 2013 — Donated to Apache Software Foundation
  • 2014 — Apache Spark 1.0 released; PySpark introduced
  • 2016 — Spark 2.0: SparkSession unifies APIs, Structured Streaming

SparkSession Configuration

What is SparkSession?

SparkSession is the unified entry point for all Spark functionality since Spark 2.0. It replaces the older SparkContext, SQLContext, and HiveContext.

    SparkSession.builder
    .appName("MyApp")
    .master("local[*]")   # use all CPU cores locally
    .getOrCreate()
)

# ── Production session ─────────────────────────────────────
spark = (
    SparkSession.builder
    .appName("ProductionETL")
    .config("spark.executor.memory",              "4g")
    .config("spark.executor.cores",               "4")
    .config("spark.driver.memory",                "2g")
    .config("spark.sql.shuffle.partitions",       "200")
    .config("spark.sql.adaptive.enabled",         "true")
    .config("spark.serializer",
            "org.apache.spark.serializer.KryoSerializer")
    .getOrCreate()
)

# Access underlying SparkContext
sc = spark.sparkContext
sc.setLogLevel("WARN")

# Read/set configs at runtime
spark.conf.set("spark.sql.shuffle.partitions", "50")
print(spark.conf.get("spark.sql.shuffle.partitions"))

# Always stop when done
spark.stop()

Spark Revolution

Understanding the Data Lake Landscape

Before HDFS & MapReduce, organisations stored data in Data Warehouses (e.g. Teradata, Exadata), pulling data in from many OLTP/OLAP systems. These warehouses faced serious limitations:

ChallengeDetail
Vertical scalingAdding more DW capacity was expensive
Capital investmentLarge upfront hardware costs
StorageNon-scalable; capacity hard to extend
Data typesSupports structured data only

To overcome these, the Data Lake concept emerged with:

  • Horizontal scaling — add cheap commodity servers to clusters
  • Low capital investment — commodity hardware & cloud storage
  • Scalable storage — cloud-based, grows on demand
  • Any data type — structured, semi-structured, and unstructured

Data Lake Architecture — 4 Core Capabilities

While storage was the original core capability, modern Data Lakes evolved 4 important capabilities:

CapabilityDescription
IngestData collection and ingestion from diverse sources
StorageDistributed, fault-tolerant, scalable storage (HDFS, S3, ADLS)
ProcessData processing and transformation (where Spark shines)
ConsumeData access and retrieval for analytics, ML, BI tools

Apache Spark & its Ecosystem

  • A unified analytics engine for large-scale distributed data processing and machine learning
  • Open-source cluster computing framework handling both batch and streaming data
  • Built on top of Hadoop MapReduce concepts but stores intermediate data in-memory — making it far faster
  • Initiated by Matei Zaharia at UC Berkeley's AMP Lab in 2009; open-sourced 2010 (BSD); Apache Top-Level Project in 2014

Spark Ecosystem — 3 Tiers

The Apache Spark ecosystem is divided into three distinct tiers:

Tier 1 — Storage & Cluster Manager

Spark is a distributed processing engine; it deliberately does not bundle its own cluster resource manager or distributed storage. This decoupling lets you plug in the combination that suits your environment:

RoleOptions
Cluster ManagerApache YARN, Apache Mesos, Kubernetes, Spark Standalone
StorageHDFS, Amazon S3, Azure Data Lake, Google Cloud Storage, Cassandra FS, and more

Tier 2 — Spark Core

The heart of Spark, containing two main components:

  • Spark Compute Engine — handles memory management, job scheduling, fault recovery, and communication with the cluster manager & storage system. Submit your job; the compute engine does the rest.
  • Spark Core APIs:
    • Structured APIs — DataFrames & Datasets, designed and optimised for structured data
    • Unstructured APIs — lower-level RDDs, Accumulators, and Broadcast variables; available in Scala, Python, Java, and R

Tier 3 — Set of Libraries

High-level libraries built on Spark Core APIs that provide specialised functionality:

LibraryPurpose
Spark SQLSQL queries & structured data processing
Spark StreamingConsume and process continuous data streams
MLlibMachine learning library with high-quality algorithms
GraphXGraph computation with a library of common graph algorithms

These libraries provide APIs, DSLs, and algorithms in multiple languages, and directly depend on Spark Core APIs to achieve distributed processing.

Hadoop to PySpark

Why PySpark?

In the early days, tools like Hadoop MapReduce were widely used to process large datasets. However, they were slow because data had to be repeatedly read from and written to disk at every stage.

PySpark (Apache Spark + Python) changed this. By storing most data in memory (RAM) instead of on disk, PySpark can run computations 10–100× faster.

What Is PySpark Used For?

PySpark is a distributed data processing framework. It allows you to:

  • Analyze very large datasets quickly that don't fit in a single machine's memory
  • Use Python-like syntax, making it accessible for developers and data scientists
  • Distribute work across multiple computers (nodes) instead of overloading a single machine

Think of it like teamwork:

  • Normal Python: One person does the entire job
  • PySpark: A team works together, each handling part of the task, then combining results

Example: Average Age Calculation

Imagine you have a huge file of client records and you need the average age:

Without PySpark (Python only)With PySpark
Input FileMay not fit in memory → crashSplit across multiple machines
ProcessingSlow, sequentialFast, parallel
ResultMay failComputed efficiently

Why Spark Came After Hadoop?

Hadoop + MapReduce worked, but had three key limitations:

  • Disk-bound — MapReduce always reads/writes from disk → slow
  • Batch-only — couldn't handle real-time streaming data
  • Tightly coupled — limited to Hadoop's HDFS/YARN ecosystem

Apache Spark was created as a faster, flexible alternative to solve all three.

Hadoop Components (for context)

  • HDFS (Hadoop Distributed File System) — stores massive files across low-cost commodity machines
  • YARN (Yet Another Resource Negotiator) — manages cluster resources and schedules jobs (MapReduce, Spark, etc.)

Spark vs Hadoop MapReduce

  • Stream Processing — process data in real time (e.g., live tweets, IoT sensors, fraud detection)
  • Data Ingestion — import data from HDFS, AWS S3, Kafka, JDBC, Twitter, and more

Unlike MapReduce (which works only with HDFS + YARN), Spark integrates seamlessly with modern ecosystems:

  • Cloud platforms — AWS (EMR, S3, Glue), Azure (HDInsight, ADLS), Google Cloud (Dataproc, GCS)
  • Databases & storage — Cassandra, MongoDB, HBase, HDFS, S3, Delta Lake

Apache Spark Runtime Architecture

What is Apache Spark?

Apache Spark is a distributed computing framework. That means:

  • Every Spark application is a distributed application
  • Distributed applications need a cluster to run efficiently

You can run Spark on your local machine for testing, but in production Spark applications run on clusters.

What is a Cluster?

A cluster is a group of networked machines (called worker nodes) that work together.

Cluster ExampleValue
Number of machines10 worker nodes
Per machine16 CPU cores + 64 GB RAM
Total capacity160 CPU cores + 640 GB RAM

Key Components

Driver

Your main program. Creates the SparkSession, translates PySpark code into tasks, requests executors, and sends tasks to them.

Cluster Manager

Handles resource allocation (CPU + memory). Most common: Hadoop YARN and Kubernetes. Also: Mesos, Spark Standalone.

Application Master

On YARN, the cluster manager launches an AM container — the "coordinator" for your Spark job. The Driver runs inside the AM.

Executor

JVM process on each worker node. Actually performs computations — runs transformations and actions on RDDs/DataFrames — and reports results back to the Driver.

Task

Smallest unit of work — processes one data partition. All tasks in a stage run in parallel.

Stage / Job

A Job is triggered by one action and consists of one or more Stages. Stages are sets of tasks separated by shuffle boundaries.

Submitting a Spark Application — Step by Step

  1. spark-submit → Cluster Manager
    The spark-submit command sends a request to the Cluster Resource Manager (YARN, Kubernetes, or Mesos) to allocate resources.
  2. Application Master Created
    On YARN, the cluster manager launches an Application Master (AM) container on one of the worker nodes. The AM manages resources and monitors execution.
  3. Driver Program Starts
    Inside the AM container, your Driver Program starts running. It translates your PySpark/Scala code into tasks and requests executors from YARN.
  4. Executors Run Tasks
    YARN allocates Executor Containers on worker nodes. Executors perform the actual computations and report results back to the Driver.
  5. Job Completion
    Once all tasks complete, results are collected by the Driver. The Application Master notifies YARN the job is done. All executor and AM containers are released.

Simplified Flow

textspark-submit flow
spark-submit
  → Cluster Manager (YARN / Kubernetes)
    → Application Master
      → Driver Program
        → Executors (worker nodes)
          → Run tasks on data partitions
            → Results returned to Driver
              → Resources released

Execution Flow (Action to Result)

  1. You call an action (e.g., df.count())
  2. Driver converts the logical plan into a DAG of stages
  3. Catalyst Optimizer rewrites and optimizes the plan
  4. DAGScheduler breaks stages into tasks
  5. TaskScheduler sends tasks to executors via the Cluster Manager
  6. Executors run tasks on their local data partitions
  7. Results are collected back to the Driver

Partitions

Data is split into partitions — the unit of parallelism. One task processes one partition.

pythonpartitions.py
# Check partition count
print(df.rdd.getNumPartitions())

# Increase partitions (full shuffle)
df = df.repartition(16)

# Partition by a column (data colocation)
df = df.repartition(8, "country")

# Reduce partitions without shuffle (use after filter)
df = df.coalesce(4)

Spark Architecture & Execution Model

Master-Slave Architecture

Apache Spark uses a master-slave architecture where the master is called the Driver and the slaves are called Workers. The master manages, maintains, and monitors the workers; the workers perform the actual processing tasks.

Spark's architecture is built on two core abstractions:

  • Resilient Distributed Dataset (RDD)
  • Directed Acyclic Graph (DAG)

RDD — The Building Block

RDD stands for:

LetterMeaningDetail
ResilientFault-tolerantCapable of rebuilding data on failure
DistributedSpread across nodesData distributed among multiple cluster nodes
DatasetPartitioned collectionCollection of partitioned data with values
  • When you load data into a Spark application, it creates an RDD that stores the loaded data
  • RDDs are immutable — they cannot be modified once created, but can be transformed into new RDDs at any time

Transformations vs Actions

TransformationsActions
What they doCreate a new RDD from an existing one without altering the originalInstruct Spark to apply computation and send the result back to the Driver
ExamplesgroupBy, map, filter, sort, renamecollect, show, write, count, save
ExecutionLazy — not executed until an action is calledTriggers execution of all pending transformations

Narrow vs Wide Transformations

Narrow TransformationsWide Transformations
PartitionsEach input partition → exactly one output partitionInput partitions → many output partitions
Shuffle?No shuffle requiredRequires shuffling data between partitions
PerformanceMore efficient — runs in parallel per partitionMore complex, more resource-intensive
Data formatsStructured, unstructured, semi-structuredTypically requires structured data
Examplesmap, filter, flatMap, uniongroupByKey, reduceByKey, join, distinct

Lazy Evaluation

Spark will not start execution until an action is called. As long as you are performing transformations, Spark just records the plan. Once an action is called, Spark looks at all pending transformations, builds a DAG, and executes it.

pythonlazy_eval.py
# These transformations are lazy — nothing runs yet
df2 = df.filter(df["age"] > 25)
df3 = df2.groupBy("country").count()

# This ACTION triggers execution of the full DAG
df3.show()   # <-- Spark builds DAG and runs here

Directed Acyclic Graph (DAG)

  • A finite directed graph representing the sequence of computations on data
  • Every Spark job creates a DAG of task stages to be executed on the cluster
  • Spark DAGs can contain many stages, unlike Hadoop MapReduce which has only two predefined stages (Map and Reduce)
  • Consecutive computation stages optimize the execution plan
  • DAGs enable fault-tolerance — if a partition is lost, Spark can recompute only that partition by replaying the DAG lineage

Spark Application Components

ComponentDescription
Spark ApplicationA program built with Spark APIs (PySpark script, Java/Scala app, spark-shell session). Consists of a Driver container and Executors.
SparkSession / SparkContextEntry point to Spark functionality. Represents the connection to the Spark cluster. Used to access lower-level APIs like RDD.
JobA parallel computation triggered by one Spark action (e.g., collect(), save()). Consists of multiple stages.
StageA set of tasks that can be executed together without a shuffle. Stages depend on each other and are separated by shuffle boundaries.
TaskThe smallest unit of work. Runs in a Spark executor, maps to a single core, and works on a single partition of data.

Driver Responsibilities

The Driver runs on the master node and is responsible for:

  • Managing SparkContext — creating and managing the main entry point of the application
  • Breaking down the application — splitting the app into parallel tasks for worker nodes
  • Scheduling tasks — assigning tasks to workers based on available resources
  • Monitoring tasks — tracking execution; rescheduling failed tasks on different nodes
  • Gathering results — collecting task results and combining them into the final output

Executor Responsibilities

The Executor runs on worker nodes and is responsible for:

  • Running tasks — executes tasks assigned by the Driver; can run multiple tasks in parallel based on available cores
  • Reporting status — sends task progress back to the Driver for monitoring
  • Storing intermediate data — caches intermediate results in memory to avoid costly shuffles between nodes
  • Releasing resources — frees CPU/memory when tasks complete for maximum efficiency

Cluster Manager Responsibilities

The Cluster Manager handles all resource management for the Spark application:

  • Resource allocation — receives resource requests from the Driver and allocates CPU cores and memory
  • Executor management — launches and manages Spark executors on worker nodes
  • Fault tolerance — monitors worker node health and reallocates resources/restarts failed tasks
  • Node management — tracks worker node status; handles registration, deregistration, and decommissioning
Cluster ManagerDescription
Spark StandaloneSimple built-in cluster manager; easy to set up
Apache MesosGeneral-purpose cluster manager; can also run Hadoop MapReduce
Hadoop YARNResource manager in Hadoop 2; most widely used in enterprise
KubernetesOpen-source container orchestration; growing standard for cloud-native Spark

Spark Execution Modes

ModeDriver LocationExecutor LocationUse Case
LocalSame JVM (your laptop)Same JVMDevelopment & testing; no cluster needed
ClientClient machine (outside cluster)Inside clusterInteractive queries; if client disconnects, job dies
ClusterInside the cluster (edge node)Inside clusterProduction; job survives client disconnect

Local Mode

Everything runs on a single JVM — no concept of nodes or distributed execution. Used for development. A single JVM produces both the Driver and the Executor.

bashterminal
# Launch local Spark shell (all cores)
pyspark --master local[*]

# Submit in local mode
spark-submit --master local[4] my_script.py

Client Mode

The Driver runs on the client machine (outside the cluster); Executors run inside the cluster. The Driver connects to the Cluster Manager, starts Executors, and receives results. If the client machine disconnects, the Driver dies and all Executors shut down.

bashterminal
spark-submit --master yarn --deploy-mode client my_script.py

Cluster Mode

Both the Driver and Executors run inside the cluster. The job is submitted from a local/edge node. If the client disconnects, the Driver is unaffected — it runs on the cluster. This is the standard mode for long-running production jobs.

bashterminal
spark-submit --master yarn --deploy-mode cluster my_script.py
Chapter 3 · Core Data Structures

RDD Fundamentals

Why RDDs? — Before Spark

Before Spark, Hadoop MapReduce dominated big data processing. It had major limitations:

  • Slow processing — every operation wrote to disk (HDFS), making iterative jobs very slow
  • Limited APIs — only map and reduce; complex workflows were difficult
  • No in-memory computation — recomputation always started from scratch if data was reused

Spark introduced RDDs to solve these problems:

  • In-memory processing — much faster than disk I/O
  • Lineage-based fault tolerance — lost data can be recomputed from the lineage graph
  • Rich transformations — map, filter, reduceByKey, join, and more
  • Ideal for iterative algorithms — ML and graph processing become significantly faster

What is an RDD?

RDD = Resilient Distributed Dataset

PropertyMeaning
DatasetCollection of records — like Python lists or Java arrays
DistributedData is split into partitions across cluster nodes for parallel processing
Resilient (Fault-Tolerant)If a partition is lost, Spark recomputes it using lineage
Type-SafeDefine RDD[Int], RDD[String], etc. to catch type errors early
Lazy EvaluationTransformations don't run until an action (count(), collect()) is triggered

RDDs support both unstructured data (logs, text files, tweets) and structured data (rows and columns).

Analogy — RDD as a Classroom

Imagine you are a teacher with 1,000 student papers to mark:

  • Checking all papers alone → takes too long
  • Divide into 10 groups → assign 10 helpers
  • Each helper marks their group in parallel
  • Results are combined and returned to you

RDD works exactly the same way: splits big data into partitions, processes them in parallel, then combines results.

Tip: Use RDDs for low-level control or unstructured data. For most modern workloads, prefer DataFrames/Datasets.

When to Use RDDs

  • Low-level transformations not supported by DataFrames
  • Unstructured data processing (logs, tweets, raw text)
  • Custom transformations requiring full control over partitioning

Creating RDDs

pythonrdd_create.py

    "data engineering with pyspark and spark",
]

text_rdd = spark.sparkContext.parallelize(data)

# Step 1 — split lines into words
words = text_rdd.flatMap(lambda line: line.split(" "))

# Step 2 — keep only words containing "spark"
spark_words = words.filter(lambda w: "spark" in w.lower())

# Step 3 — count occurrences
counts = spark_words.map(lambda x: (x, 1)).reduceByKey(lambda a, b: a + b)

print(counts.collect())

spark.stop()
textoutput
[('pyspark', 3), ('spark', 2)]

How it Executes — Step by Step

  1. Load Big Data — Spark reads the dataset (textFile / parallelize)
  2. Split into Partitions — data is divided across multiple nodes
  3. Transformations (Lazy) — map(), filter(), flatMap() applied per partition but not executed yet
  4. Action Triggers Execution — collect(), count(), take() run all pending transformations in parallel
  5. Fault Tolerance — if a partition is lost, Spark recomputes it using the recorded lineage

Common Transformations

pythonrdd_transforms.py
words = sc.parallelize(["hello world", "hello spark", "pyspark guide"])

# map — 1-to-1 transform
upper = words.map(lambda x: x.upper())

# flatMap — 1-to-many + flatten
word_rdd = words.flatMap(lambda x: x.split())

# filter
h_words = word_rdd.filter(lambda w: w.startswith("h"))

# reduceByKey — classic MapReduce pattern
counts = (
    words.flatMap(lambda line: line.split())
         .map(lambda w: (w, 1))
         .reduceByKey(lambda a, b: a + b)
)

# mapPartitions — one call per partition (efficient for DB connections)
def process(iterator):
    yield from (w.upper() for w in iterator)

word_rdd.mapPartitions(process)

Limitations of RDDs

  • Spark cannot optimize inside custom functions (e.g., lambdas) — no Catalyst optimizer
  • Inefficient operations if not careful (e.g., filtering after a wide shuffle)
  • Debugging large distributed RDD pipelines is harder than DataFrames
Modern Spark tip: Prefer DataFrames/Datasets. Use RDDs only for low-level or unstructured data needs. Understanding RDDs helps you understand how Spark works internally.

Understanding RDD in Apache Spark

Key Features of RDD

#FeatureDetail
1ImmutabilityOnce created, an RDD cannot be changed — ensures consistency across distributed nodes
2DistributedData is divided into partitions and processed in parallel across the cluster
3Fault ToleranceIf any partition is lost, Spark recomputes it from the lineage graph
4In-Memory ComputationData is stored in memory for fast processing — avoids disk I/O between stages
5Lazy EvaluationTransformations form a lineage (DAG) and only execute when an action is triggered
6PersistenceRDDs can be cached in memory or stored on disk for reuse across multiple actions

RDD Operations Reference

Transformations (create a new RDD, lazy)

TransformationDescription
map(f)Apply function f to each element; returns one element per input
filter(f)Keep only elements where f returns True
flatMap(f)Like map but flattens the result (one element → zero or more)
distinct()Remove duplicate elements
union(other)Return elements from both RDDs (with duplicates)
intersection(other)Return only elements present in both RDDs
subtract(other)Return elements in this RDD but not in other
reduceByKey(f)Merge values for each key using function f
groupByKey()Group values by key (use reduceByKey when possible — more efficient)

Actions (trigger execution, return result)

ActionDescription
collect()Return all elements to the driver (use only for small RDDs)
count()Count the number of elements
first()Return the first element
take(n)Return the first n elements
reduce(f)Aggregate all elements using function f
saveAsTextFile(path)Save RDD contents to external storage (HDFS, local)

Internal Working of RDD

  1. When you apply transformations, Spark builds a DAG — nothing executes immediately, it just records the lineage
  2. When you call an action, Spark’s scheduler breaks the DAG into stages and tasks
  3. Tasks are distributed across cluster nodes and executed in parallel
  4. Fault tolerance: if a node fails, missing partitions are recomputed by replaying the lineage

Advantages of RDD

  • Fast processing via in-memory computation
  • Easy fault recovery using lineage — no need to reprocess entire dataset
  • Supports functional programming style (map, filter, reduce)
  • Works on both structured and unstructured data
  • Flexible for low-level, fine-grained transformations

Limitations of RDD

  • Verbose code compared to DataFrames/Datasets
  • No automatic optimization — Catalyst optimizer only works with DataFrames
  • Inefficient for structured data — DataFrames handle schema-aware data far better

When to Use RDD

  • Fine-grained transformations not supported by DataFrames
  • Unstructured data (logs, raw text, binary data)
  • Low-level control over data distribution and computation

Introduction to PySpark DataFrames

What is a DataFrame?

Think of a PySpark DataFrame as a giant, distributed spreadsheet. According to Spark documentation, a DataFrame is a dataset organized into named columns — much like a table in a relational database or a DataFrame in Python/R. The key difference is that this "spreadsheet" is automatically split across multiple machines, so you can work with gigabytes or terabytes of data without running out of memory.

Rows, Columns, and Table Analogy

  • Rows — each row is a record in your dataset
  • Columns — each column represents a field or attribute
  • DataFrame — the overall table combining rows and columns into a structured dataset

Because PySpark DataFrames are distributed, data is broken into partitions. Each partition is processed in parallel by different executors, allowing Spark to handle data that would overwhelm a single machine.

PySpark vs Pandas DataFrames

FeaturePandas DataFramePySpark DataFrame
Execution environmentLocal machine (single process)Distributed cluster
Data sizeMB to a few GB (must fit in RAM)GB to TB (distributed across nodes)
PerformanceCan slow down / OOM on large dataBuilt for big data — parallelizes work
API stylePythonic and flexibleSQL-like and functional
MutabilityMutable in-placeImmutable — every transform creates a new DF
Lazy evaluationEager (runs immediately)Lazy (builds DAG, runs on action)

When to Use Each

  • Choose Pandas for exploratory analysis or projects where data fits comfortably in RAM and you want a quick, interactive workflow
  • Choose PySpark for ETL jobs, large-scale data engineering, or ML tasks that require distributed processing

from pyspark.sql.functions import broadcast

emp  = spark.read.parquet("data/employees")
dept = spark.read.parquet("data/departments")

# Inner join on shared column name
result = emp.join(dept, on="dept_id", how="inner")

# Left join with compound condition
result = emp.join(dept,
    (emp.dept_id == dept.dept_id) & (emp.region == dept.region),
    how="left")

result = emp.join(broadcast(dept), on="dept_id")

# Resolve ambiguous columns with aliases
e = emp.alias("e")
d = dept.alias("d")
result = e.join(d, col("e.dept_id") == col("d.dept_id"))

Window Functions

Window functions compute values across rows related to the current row without collapsing the DataFrame.

pythonwindow.py
from pyspark.sql import Window
from pyspark.sql import functions as F

# ── Define window specs ────────────────────────────────────
win_dept = (Window.partitionBy("department").orderBy(col("salary").desc()))
win_all  =  Window.partitionBy("department")

# ── Ranking ────────────────────────────────────────────────
df.withColumn("rank",       F.rank().over(win_dept))       # gaps on ties: 1,1,3
df.withColumn("dense_rank", F.dense_rank().over(win_dept)) # no gaps: 1,1,2
df.withColumn("row_num",    F.row_number().over(win_dept)) # always unique
df.withColumn("ntile",     F.ntile(4).over(win_dept))     # quartiles

# ── Aggregate over window ─────────────────────────────────
df.withColumn("dept_avg",   F.avg("salary").over(win_all))
df.withColumn("dept_total", F.sum("salary").over(win_all))
df.withColumn("pct",        (col("salary") / F.sum("salary").over(win_all)))

# ── Running total ─────────────────────────────────────────
win_run = (Window.partitionBy("department").orderBy("hire_date")
           .rowsBetween(Window.unboundedPreceding, Window.currentRow))
df.withColumn("cumulative", F.sum("salary").over(win_run))

# ── Lag / Lead ────────────────────────────────────────────
win_t = Window.partitionBy("user_id").orderBy("event_ts")
df.withColumn("prev", F.lag("event", 1).over(win_t))
df.withColumn("next", F.lead("event", 1).over(win_t))

# ── Top-N per group ───────────────────────────────────────
top2 = (df.withColumn("rn", F.row_number().over(win_dept))
          .filter(col("rn") <= 2).drop("rn"))

PySpark Structured Streaming

Structured Streaming treats a live stream as an unbounded table. New data arrives as appended rows.

pythonstreaming.py
from pyspark.sql import functions as F
from pyspark.sql.types import StructType, StructField, StringType, LongType

# ── Read from Kafka ────────────────────────────────────────
raw = (spark.readStream
       .format("kafka")
       .option("kafka.bootstrap.servers", "broker:9092")
       .option("subscribe", "events")
       .option("startingOffsets", "latest")
       .load())

event_schema = StructType([
    StructField("user_id",    LongType()),
    StructField("event_type", StringType()),
    StructField("ts",         LongType()),
])
parsed = (raw
    .select(F.from_json(col("value").cast("string"), event_schema).alias("d"))
    .select("d.*")
    .withColumn("event_time", (F.col("ts") / 1000).cast("timestamp")))

# ── Windowed aggregation with watermark ───────────────────
windowed = (parsed
    .withWatermark("event_time", "10 minutes")
    .groupBy(F.window("event_time", "5 minutes", "1 minute"), "event_type")
    .agg(F.count("*").alias("event_count")))

# ── Sink to Delta ─────────────────────────────────────────
query = (windowed.writeStream
    .format("delta")
    .outputMode("append")
    .option("checkpointLocation", "s3a://bucket/checkpoints/events")
    .trigger(processingTime="30 seconds")
    .start("s3a://bucket/tables/event_counts"))

query.awaitTermination()

# ── foreachBatch — arbitrary logic per micro-batch ────────
def process_batch(batch_df, batch_id):
    batch_df.write.mode("append").parquet(f"output/batch_{batch_id}")

(parsed.writeStream.foreachBatch(process_batch)
       .option("checkpointLocation", "/tmp/ckpt").start())
Output ModeWhen to Use
appendNew rows only — no updates to previous output. Default.
completeFull result table every trigger. For aggregations without watermark.
updateOnly changed rows since last trigger.

Chapter 8 · Interview Questions

PySpark Interview Questions

Fundamentals

Q: What is the difference between a transformation and an action? Transformations (filter, select, map) are lazy — they build a logical plan (DAG) but don't execute. Actions (count, show, collect, write) trigger execution of the full plan.
Q: What is a DAG in Spark? A Directed Acyclic Graph — the logical execution plan Spark builds from your transformations. Spark's Catalyst Optimizer rewrites it before physical execution, applying rule-based and cost-based optimizations.
Q: What is the difference between repartition() and coalesce()? repartition(n) performs a full shuffle and can increase or decrease partitions. coalesce(n) only decreases partitions and avoids a full shuffle — use it when reducing partition count.
Q: What is the difference between cache() and persist()? cache() is equivalent to persist(StorageLevel.MEMORY_AND_DISK). persist() lets you choose the storage level: MEMORY_ONLY, MEMORY_AND_DISK, DISK_ONLY, etc.
Q: What causes a shuffle and why is it expensive? A shuffle occurs when data must be redistributed across partitions — triggered by groupBy, join, orderBy, distinct. It involves disk I/O, network transfer, and serialization — the most expensive Spark operation.

Performance

Q: How do you handle data skew? Salting: add a random salt column (0–N) to the skewed key, explode the small side with all salt values, then join on the salted key. AQE's skewJoin optimization can also handle this automatically in Spark 3.x.
Q: When should you use broadcast join? When one side of the join is small enough to fit in executor memory (typically <100MB). Spark replicates it to all executors, eliminating the shuffle. Set spark.sql.autoBroadcastJoinThreshold or use broadcast() hint.
Q: What is predicate pushdown? Spark pushes filter conditions down to the data source (e.g., Parquet reader, JDBC query) so that only matching rows are read — reducing I/O significantly.

Architecture

Q: What is the role of the Catalyst Optimizer? Catalyst is Spark's query optimization engine. It applies rule-based transformations (predicate pushdown, column pruning, constant folding) and cost-based optimizations to produce an efficient physical execution plan.
Q: What is Tungsten? Tungsten is Spark's execution engine that uses off-heap memory management, cache-aware algorithms, and whole-stage code generation (compiling query plans to JVM bytecode) for near-native CPU performance.
Q: What is the difference between narrow and wide transformations? Narrow: each input partition maps to exactly one output partition — no shuffle (map, filter, select). Wide: input partitions may contribute to multiple output partitions — requires a shuffle (groupBy, join, repartition).

Streaming

Q: What is a watermark in Structured Streaming? A watermark tells Spark how long to wait for late-arriving data. E.g., withWatermark("event_time", "10 minutes") means Spark will wait 10 minutes beyond the max seen event time before closing a window.
Q: What is exactly-once processing in Structured Streaming? Spark Structured Streaming guarantees exactly-once semantics when using supported sources (Kafka, files) and sinks (Delta Lake, Kafka) with checkpointing enabled. The checkpoint tracks offsets and committed state.
